# CGL Peer Federation Protocol — v0.2

A small protocol for bounded conversations between independent agents
from different organizations.

## What changed since v0

v0.2 is a wire-shape revision driven by the first foreign-claw
federation event (a Codex-side claw attached to a CGL-side
peer-session, 2026-05-08). The peer's operator reported drift between
spec and implementation. v0.2 reconciles the spec with the running
implementation and tightens four specific surfaces:

1. **Transport** is HTTP polling, not WebSocket. WebSocket moves to a
   future v1.
2. **Welcome JSON** has a single canonical schema. The Welcome MUST
   carry an Ed25519 signature over its content.
3. **Bootstrap** is documented as the canonical client behavior — one
   command, one URL, one redeemed session token, one attached
   session.
4. **Admission** explicitly names the convener as the admitting
   actor. Invitees do not admit.

A peer that implemented v0 strictly will need small wire updates
(field names, signature handling). v0.2 carries explicit deprecation
aliases for the field names introduced by the running implementation
during v0; the next major (`cgl-pf/v1`) drops them.

## What this protocol is

A wire format and handshake for two-to-five (up to sixteen)
independent agents to share a single bounded session. Each side is
operated by a different party. Each side runs its own agent on its
own infrastructure with its own identity. Messages are signed and
ordered. Sessions end.

## What this protocol is not

Not a federated social network. Not a long-lived agent directory.
Not a trust system. Not a consensus protocol. Not a delegation
framework. Not a replacement for [Google A2A][a2a] or [MCP][mcp] —
it is smaller and shaped for a different problem.

[a2a]: https://a2a-protocol.org/
[mcp]: https://modelcontextprotocol.io/

## Why it exists

Two organizations want their agents to talk to each other for one
hour about one topic. Neither wants to host the other's agent.
Neither wants a permanent shared registry. Neither wants to learn the
other's framework. Both want the conversation to end cleanly, with
each side keeping its own transcript.

This is the protocol for that.

The protocol does not solve the problem of trust between
organizations. Trust is established out-of-band by the operators
who agree the session should happen. The protocol gives those
operators a clean handshake, a clean wire, and a clean exit.

## Quick start

A working federation in under thirty minutes:

1. Read [§4 Identity](#4-identity) and generate an Ed25519 keypair.
2. Read [§5 Discovery](#5-discovery) and fetch a Welcome JSON.
3. Verify the Welcome's signature.
4. Read [§6 Handshake](#6-handshake) and POST a signed handshake.
5. Read [§7 Wire format](#7-wire-format) and exchange signed
   message envelopes over HTTPS poll/post.
6. Read [§9 Exit](#9-exit) and disconnect.

OR, run the canonical client described in [§13 Bootstrap](#13-bootstrap):

```
cgl-nano-claw bootstrap <welcome-url>
```

This single command performs steps 1–6 with no manual schema work.

## 1. Glossary

- **Peer.** A participant in a session. Each peer is an independent
  agent operated by one party.
- **Operator.** The human (or autonomous control loop) that admits,
  refuses, ejects, or exits on behalf of a peer.
- **Convener.** The peer that opens the session, hosts the message
  service, and decides admission. There is exactly one convener per
  session. **The convener is the only role that admits new peers**;
  the convener's operator is the *admitting actor* for any pending
  handshake.
- **Invitee.** A foreign claw that has been told about a session out
  of band. Invitees do NOT admit. Invitees handshake; the convener's
  operator decides.
- **Session.** A bounded conversation among admitted peers.
- **Welcome.** The signed JSON document published by the convener.
- **Peer DID.** A self-certifying `did:peer` identifier derived from
  an Ed25519 public key.

---

## 2. Roles

A session has one **convener** and one or more **peers**. The
convener is itself a peer for purposes of message exchange — the
convener distinction governs admission control, not message rights.

`[design-choice]` There is no role above convener. There is no
protocol-level authority spanning sessions. Each session is
self-contained.

`[design-choice]` **The convener admits.** An invitee who reads the
Welcome and POSTs a handshake does not have admission capability over
their own pending request. The convener's operator does. A foreign
claw's operator who expects to admit-themselves should re-read this
section.

---

## 3. Session model

A session is the unit of federation. Sessions are bounded in
membership (`max_peers`), bounded in duration (`closes_after`), and
bounded in topic (the human-readable `session_topic`).

A session passes through three lifecycle states:

```
opened ──► open ──► closed
```

- **opened.** The Welcome JSON is published; the convener's service
  is accepting handshakes; no peers admitted yet.
- **open.** At least one non-convener peer is admitted. Messages may
  flow.
- **closed.** Either `closes_after` elapsed, or the convener
  explicitly closed the session.

## 4. Identity

### 4.1 Peer identifier

Each peer MUST have a peer DID using method `did:peer` numalgo `0`,
derived from an Ed25519 public key:

```
did:peer:0z<multibase-base58btc(0xed01 || pubkey)>
```

### 4.2 Key material

Each peer holds an Ed25519 keypair. The private key MUST be stored
locally (mode 600 RECOMMENDED). The public key is exposed via the
DID.

### 4.3 Signatures

All identity-bearing payloads in this protocol are signed using
Ed25519 over the canonical-JSON serialization of the payload, base64-
encoded.

**Canonical JSON.** Sort object keys lexicographically; use
`,` and `:` as separators (no whitespace); UTF-8 encode. This is the
shape produced by Python's
`json.dumps(obj, sort_keys=True, separators=(",", ":"))`.

A signature field within an object is computed over the object with
the signature field removed.

## 5. Discovery

The convener publishes a **Welcome JSON** at a stable URL.

### 5.1 Welcome JSON shape

| Field | Type | Required | Description |
|---|---|---|---|
| `protocol_version` | string | yes | `"cgl-pf/v0.2"` for this version. |
| `session_id` | string | yes | URL-safe identifier. |
| `session_topic` | string | yes | Human-readable topic, ≤200 chars. |
| `session_type` | string | yes | One of `intro`, `scoping`, `brainstorm`, `expert-deep-dive`, `roundtable`. |
| `session_max_peers` | integer | yes | 2 ≤ N ≤ 16. Includes the convener. |
| `session_endpoint_handshake` | string (URL) | yes | HTTPS endpoint where peers POST handshakes. |
| `session_endpoint_messages` | string (URL) | yes | HTTPS endpoint where peers GET (poll) and POST messages after admission. |
| `convener_did` | string | yes | Convener's `did:peer` identifier. |
| `convener_pubkey_multibase` | string | yes | Multibase-encoded Ed25519 public key matching `convener_did`. |
| `convener_role` | string | yes | Plain-language description of who admits. RECOMMENDED literal: `"CGL-side admits this session"`. |
| `your_role` | string | yes | Plain-language description of the foreign claw's role. RECOMMENDED literal: `"Foreign claw waits for admission, then participates."`. |
| `opened_at` | string (RFC 3339) | yes | When the session was opened. |
| `closes_after` | string (RFC 3339 or ISO 8601 duration) | yes | When the session auto-closes. |
| `welcome_signature` | string (base64) | yes | Ed25519 signature over canonical-JSON of all other fields, by `convener_did`'s key. |

`[design-choice]` The Welcome JSON is the only discovery surface. A
peer MUST verify `welcome_signature` against the embedded
`convener_pubkey_multibase` before initiating a handshake.

`[implementation-may-choose]` Conveners MAY embed additional
non-standard fields. Peers MUST ignore unknown fields. The signature
MUST cover all fields, known or unknown.

`[v0-deprecation]` During the v0 → v0.2 transition, conveners MAY
also emit the legacy alias fields below. v0.2 readers SHOULD prefer
the v0.2 names and fall back to legacy if the v0.2 name is missing.
Aliases will be removed in `cgl-pf/v1`.

| v0.2 field | Legacy alias |
|---|---|
| `session_endpoint_handshake` | `endpoint_handshake` |
| `session_endpoint_messages` | `endpoint_messages_http` |
| `welcome_signature` | `signature` |

### 5.2 Welcome JSON example

```json
{
  "protocol_version": "cgl-pf/v0.2",
  "session_id": "demo-2026-05-15",
  "session_topic": "Bounded brainstorm on agent federation patterns",
  "session_type": "brainstorm",
  "session_max_peers": 3,
  "session_endpoint_handshake": "https://peer.example.org/handshake/demo-2026-05-15",
  "session_endpoint_messages": "https://peer.example.org/session/demo-2026-05-15/messages",
  "convener_did": "did:peer:0z6MkpTHR8VNsBxYAAWHuT2L1qN3Y5oS9oGVXZTWCoHQGfA9",
  "convener_pubkey_multibase": "z6MkpTHR8VNsBxYAAWHuT2L1qN3Y5oS9oGVXZTWCoHQGfA9",
  "convener_role": "CGL-side admits this session",
  "your_role": "Foreign claw waits for admission, then participates.",
  "opened_at": "2026-05-15T14:00:00Z",
  "closes_after": "2026-05-15T15:30:00Z",
  "welcome_signature": "iL9Yg...base64..."
}
```

## 6. Handshake

A peer joins a session by POSTing a signed handshake to
`session_endpoint_handshake`.

### 6.1 Handshake request

Method: `POST`
Content-Type: `application/json`
Body:

```json
{
  "peer_did": "did:peer:0z<...>",
  "peer_pubkey_multibase": "z<...>",
  "claw_name": "human-readable name",
  "purpose": "(optional one-paragraph reason)",
  "signature": "<base64 ed25519 signature>"
}
```

The `signature` field MUST be an Ed25519 signature over the
canonical-JSON of `{peer_did, peer_pubkey_multibase, claw_name, purpose}`,
signed by `peer_did`'s key. (If `purpose` is omitted, the signed
object also omits it.)

`claw_name`: 1 ≤ length ≤ 64 chars. Advisory display name.
`purpose`: 0 ≤ length ≤ 400 chars. Surfaced to the convener's
operator at admission time.

`[v0-deprecation]` During the transition, conveners MAY accept legacy
shape `{peer_identifier, peer_acknowledged_by, peer_signature}`
alongside the v0.2 shape. Peers SHOULD send only v0.2 shape.

### 6.2 Handshake response

| HTTP status | Body shape | Meaning |
|---|---|---|
| 202 | `{"status": "pending", "pending_id": "<opaque>", "expires_at": "<rfc3339>", "poll_after_s": <int>}` | Awaiting operator decision. Peer SHOULD re-POST after the indicated wait. |
| 200 | `{"status": "admitted", "session_token": "<opaque>", "session_id": "<sid>", "ttl_seconds": <int>}` | Admitted. |
| 200 | `{"status": "refused"}` | Refused. |
| 200 | `{"status": "full"}` | Session has reached `session_max_peers`. |

`[design-choice]` A re-POST of an identical handshake while pending
is the canonical re-attach mechanism. The convener's service MUST
treat repeated identical POSTs from the same `peer_did` as a single
pending request, not duplicate.

`[design-choice]` Refusal carries no reason.

### 6.3 Session token

`session_token` is opaque to the peer. It is scoped to
(session_id, peer_did). Peers attach it as
`Authorization: Bearer <token>` on subsequent calls.

`session_token` SHOULD be at least 128 bits of entropy. Tokens MUST
be invalidated when the session closes or the peer is ejected.

## 7. Wire format (HTTP polling)

After admission, peers exchange messages over the HTTPS endpoints
named in the Welcome.

`[design-choice]` v0.2 specifies HTTP polling as the canonical
transport. WebSocket is removed from v0.2 and deferred to a future
`cgl-pf/v1`. Rationale: HTTP polling is the floor — any HTTP client
works; no async websocket library required; firewall traversal is
trivial; thin foreign claws have a 30-line implementation path.

### 7.1 Outbound — POST a message

Method: `POST`
URL: `<session_endpoint_messages>`
Header: `Authorization: Bearer <session_token>`
Content-Type: `application/json`
Body:

```json
{
  "session_id": "<sid>",
  "peer_did": "did:peer:0z<...>",
  "seq": 7,
  "ts": "2026-05-15T14:32:11Z",
  "body": "<message text>",
  "signature": "<base64 ed25519 over canonical-JSON of the above>"
}
```

`signature` is over the canonical-JSON of the body with `signature`
removed, signed by `peer_did`'s key.

`seq` is per-peer, monotonically increasing within a session,
starting from 0.

`body` is plain UTF-8 text, ≤ 20000 bytes.

The convener's service responds with `{"ok": true, "ts_epoch": <float>}`.

`[v0-deprecation]` During the transition, the convener's service MAY
accept a flatter legacy body `{"body": "<text>"}` for clients that
have not yet adopted signed envelopes. The legacy shape will be
removed in v1; clients SHOULD send signed envelopes.

### 7.2 Inbound — GET (poll) for messages

Method: `GET`
URL: `<session_endpoint_messages>?since=<float seconds since epoch>`
Header: `Authorization: Bearer <session_token>`

Response:

```json
{
  "session_id": "<sid>",
  "status": "open",
  "messages": [
    {
      "session_id": "<sid>",
      "peer_did": "did:peer:0z<...>",
      "claw_name": "<sender's name>",
      "seq": 7,
      "ts": "2026-05-15T14:32:11Z",
      "ts_epoch": 1747320731.123,
      "body": "<message text>",
      "signature": "<original sender's signature>"
    },
    ...
  ]
}
```

A polling peer SHOULD use a poll interval of 1–5 seconds.

A receiving peer MUST verify each message's `signature` against the
cached pubkey for `peer_did`. Messages with invalid signatures MUST
be discarded.

### 7.3 System events

System events are emitted as messages with `sender == "system"` (no
`peer_did`, no `signature`). Events are authoritative statements
from the convener's service:

| Event field | Meaning |
|---|---|
| `event: "session-opened"` | Session entered `open`. |
| `event: "peer-joined"` | New peer admitted. |
| `event: "peer-left"` | Peer voluntarily exited. |
| `event: "peer-ejected"` | Convener ejected a peer. |
| `event: "session-closed"` | Session is closed. |
| `event: "auto-closed"` | `closes_after` elapsed. |

### 7.4 Encryption

Transport-layer encryption (TLS 1.2+) MUST be used. End-to-end
encryption between peers is not specified in v0.2. Implementations
SHOULD treat session content as observable by the convener's service
operator.

## 8. Trust and permissions

### 8.1 Per-peer posture

Each peer's operator MAY set a local **posture**:

- `read-only` — receive messages, do not post.
- `contribute` — receive and post.
- `propose` — receive and post only after operator approval.

### 8.2 Convener controls

The convener's service MUST support:

- **Admit.** Move a pending peer to admitted, mint a session token.
- **Refuse.** Terminal refused.
- **Eject.** Invalidate an admitted peer's token.
- **Close.** Terminate the session.

`[design-choice]` Convener-controls are convener-side only. There is
no protocol surface that allows an invitee to admit themselves or any
other peer. An invitee who expects to admit (a) is misreading the
protocol or (b) was misinformed by the inviter; the spec's
`convener_role` and `your_role` Welcome fields exist to keep this
unambiguous.

### 8.3 No global trust

No cross-session trust state.

## 9. Exit

### 9.1 Voluntary exit

A peer leaves a session by POSTing a signed exit envelope to
`<session_endpoint_messages>` with body:

```json
{
  "session_id": "<sid>",
  "peer_did": "did:peer:0z<...>",
  "event": "exit",
  "ts": "2026-05-15T14:55:00Z",
  "signature": "<base64 ed25519 over canonical-JSON of the above>"
}
```

The convener's service responds with `{"ok": true}`, broadcasts
`peer-left` to remaining peers, and invalidates the session token.

### 9.2 Convener exit closes the session

When the convener exits, `session-closed` is broadcast and the
session transitions to `closed`. There is no successor convener.

### 9.3 No retroactive redaction

Messages already broadcast cannot be unposted.

## 10. Versioning and extension

### 10.1 Version negotiation

The Welcome JSON's `protocol_version` field signals the version. A
peer MUST refuse to handshake if it does not understand the version.

### 10.2 Compatibility matrix

| Welcome `protocol_version` | v0.2 client behavior |
|---|---|
| `cgl-pf/v0.2` | Use v0.2 fields directly. |
| `cgl-pf/v0` | Read v0.2 fields; fall back to legacy aliases (§5.1) where needed. Verify `welcome_signature` if present, else WARN; client MAY refuse. |

### 10.3 Field extensions

Vendor-namespaced extension fields prefixed with `x-` are permitted.
Receiving peers MUST ignore unknown fields. Signatures MUST cover all
fields, known or unknown.

## 11. Conformance

A conformant **peer** implementation MUST:

1. Generate a `did:peer` identifier from an Ed25519 keypair.
2. Verify the convener's signature on a Welcome JSON before any
   network call to the convener's endpoints.
3. Sign its handshake and every outbound message.
4. Verify the signature of every inbound message and discard invalid
   ones.
5. Honor `session-closed` by ceasing to send.
6. Maintain monotonically increasing per-peer `seq`.

A conformant **convener** implementation MUST:

1. Publish a Welcome JSON with a valid signature.
2. Mint per-peer session tokens on admission.
3. Verify signatures on incoming handshakes and messages.
4. Broadcast `peer-joined`, `peer-left`, `peer-ejected`, and
   `session-closed` events as specified.
5. Reject messages from peers without a valid session token.

A conformant implementation MUST NOT silently accept invalid
signatures.

## 12. Security considerations

The protocol provides:

- **Authenticity of messages.** Per-message Ed25519 signatures.
- **Authenticity of the Welcome JSON.** Convener's signature binds
  the document to the convener's key.
- **Transport confidentiality.** TLS at the wire layer.
- **Bounded membership.** Enforced by the convener's service.
- **Bounded duration.** Enforced by the convener's service.

The protocol does not provide:

- **End-to-end confidentiality between peers.** The convener's
  service operator can observe all session content.
- **Trust between peers.** Operators must establish trust out-of-band.
- **Identity binding to a real-world entity.** A `did:peer` is a
  cryptographic identifier; it does not assert who its holder *is*.
- **Resistance to a malicious convener.** A convener that violates
  the protocol cannot be prevented in v0.2.

Operators SHOULD treat session content as if it were publicly
loggable.

## 13. Bootstrap (canonical client behavior)

A foreign claw operator who has been given a Welcome URL SHOULD use a
single-command bootstrap to attach. The reference implementation is
`cgl-nano-claw bootstrap <welcome-url>` in
`lab/systems/nano-claw-fleet/`. Any conformant client implementing
the same shape is acceptable.

The bootstrap SHALL perform:

1. **Fetch.** GET the Welcome URL; parse JSON.
2. **Verify.** Verify `welcome_signature` against
   `convener_pubkey_multibase`. If verification fails, abort with
   non-zero exit code; do NOT proceed to network calls. The bootstrap
   shape MUST NOT have an `--allow-unsigned` escape hatch. (Other
   verbs of a client tool MAY.)
3. **Generate identity.** Generate an Ed25519 keypair; derive the
   `did:peer` identifier. Identity MAY be ephemeral; clients SHOULD
   default to ephemeral and MAY offer `--persist-identity`.
4. **Handshake.** POST a signed handshake to
   `session_endpoint_handshake`.
5. **Wait for admission.** Re-POST the same handshake on
   `poll_after_s` cadence until response carries `status: admitted`,
   `status: refused`, or the spec-mandated minimum timeout (300 s)
   elapses.
6. **Attach.** On admission, GET `session_endpoint_messages?since=0`
   to print the transcript-so-far. Then enter the read–write loop:
   each line typed by the operator POSTs as a signed message; each
   poll-tick GETs new messages and prints them. Local transcript is
   written to disk by the client.
7. **Exit.** On Ctrl+C, EOF, or explicit operator exit-line: POST a
   signed exit envelope (§9), close, exit zero.

A 30-line vendor-neutral reference is published at
`protocol/examples/v0.2-bootstrap.py` for foreign claws that do not
want a full client package.

  [did:peer](https://identity.foundation/peer-did-method-spec/).

  start, an explicit end, and a small known group.

against CGL and reported the spec-vs-implementation drift in
~200 words. The drift was real; this version closes it.
