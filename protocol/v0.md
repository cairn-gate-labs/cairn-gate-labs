# CGL Peer Federation Protocol — v0

A small protocol for bounded conversations between independent agents
from different organizations.

## What this protocol is

A wire format and handshake for two-to-five independent agents to
share a single bounded session. Each side is operated by a different
party. Each side runs its own agent on its own infrastructure with
its own identity. Messages are signed and ordered. Sessions end.

## What this protocol is not

Not a federated social network. Not a long-lived agent directory.
Not a trust system. Not a consensus protocol. Not a delegation
framework. Not a replacement for [Google A2A][a2a] or
[MCP][mcp] — it is smaller and shaped for a different problem.

[a2a]: https://a2a-protocol.org/
[mcp]: https://modelcontextprotocol.io/

## Why it exists

Two organizations want their agents to talk to each other for one
hour about one topic. Neither wants to host the other's agent.
Neither wants a permanent shared registry. Neither wants to learn the
other's framework. Both want the conversation to end cleanly, with
each side keeping its own transcript.

This is the protocol for that.

The protocol does not solve the problem of trust between
organizations. Trust is established out-of-band by the operators
who agree the session should happen. The protocol gives those
operators a clean handshake, a clean wire, and a clean exit.

## Quick start

A working federation in under two hours:

1. Read [§4 Identity](#4-identity) and generate an Ed25519 keypair.
2. Read [§5 Discovery](#5-discovery) and fetch a Welcome JSON from
   a peer's session URL.
3. Read [§6 Handshake](#6-handshake) and POST a signed handshake.
4. Read [§7 Wire format](#7-wire-format) and exchange JSON-RPC
   messages over WebSocket.
5. Read [§9 Exit](#9-exit) and disconnect.

See [`example-handshake.md`](example-handshake.md) for a 200-line
worked example in pseudocode — pure HTTP + WebSocket, no framework.

## 1. Glossary

- **Peer.** A participant in a session. Each peer is an independent
  agent operated by one party.
- **Operator.** The human (or autonomous control loop) that admits,
  refuses, ejects, or exits on behalf of a peer.
- **Convener.** The peer that opens the session, hosts the message
  service, and decides admission. There is exactly one convener per
  session.
- **Session.** A bounded conversation among admitted peers. A session
  has a single message log, a closing time, and a defined set of
  admitted peers.
- **Welcome.** The JSON document published by the convener that
  describes the session and gives peers everything they need to
  initiate a handshake.
- **Peer DID.** A self-certifying identifier, of method `did:peer`,
  derived from a peer's Ed25519 public key.

---

## 2. Roles

A session has one **convener** and one or more **peers**. The
convener is itself a peer for purposes of message exchange — the
convener distinction governs admission control, not message rights.

`[design-choice]` There is no role above convener. There is no
protocol-level authority spanning sessions. Each session is
self-contained.

---

## 3. Session model

A session is the unit of federation. Sessions are bounded in
membership (`max_peers`), bounded in duration (`closes_after`), and
bounded in topic (the human-readable `session_topic`).

A session has exactly one canonical message log. The convener's
service is the authoritative orderer of messages across peers.

`[design-choice]` Each peer keeps a local copy of the transcript.
There is no shared archive across peers. The convener may retain a
canonical copy on its service per its own retention policy; the
protocol does not require the convener to expose this archive to
other peers after the session ends.

A session passes through three lifecycle states:

```
opened ──► open ──► closed
```

- **opened.** The Welcome JSON is published; the convener's service
  is accepting handshakes; no peers admitted yet.
- **open.** At least one non-convener peer is admitted. Messages may
  flow.
- **closed.** Either `closes_after` elapsed, or the convener
  explicitly closed the session. New messages MUST NOT be accepted.
  Existing peers' local transcripts persist.

## 4. Identity

### 4.1 Peer identifier

Each peer MUST have a peer DID using method `did:peer` numalgo `0`,
derived from an Ed25519 public key. The encoding is:

```
did:peer:0z<multibase-base58btc(0xed01 || pubkey)>
```

Where `0xed01` is the multicodec prefix for Ed25519 public keys and
`||` is concatenation.

`[design-choice]` `did:peer` is the floor. A peer with a stable
public web presence MAY also publish a `did:web` document that
references the same key; peers cache by DID, not by URL.

A peer DID is generated once per peer at first use. It is reused
across sessions.

### 4.2 Key material

Each peer holds an Ed25519 keypair. The private key MUST be stored
locally (mode 600 on POSIX systems is RECOMMENDED). The public key
is exposed via the DID.

A peer MAY rotate keys, in which case its DID changes. The protocol
does not specify a rotation mechanism — a rotated peer presents a
new DID and re-introduces itself out-of-band.

### 4.3 Signatures

All identity-bearing payloads in this protocol are signed using
Ed25519 over the canonical-JSON serialization of the payload, base64-
encoded.

**Canonical JSON.** Sort object keys lexicographically; use
`,` and `:` as separators (no whitespace); UTF-8 encode. This is the
shape produced by Python's `json.dumps(obj, sort_keys=True,
separators=(",", ":"))`.

A signature field within an object is computed over the object with
the signature field removed.

## 5. Discovery

The convener publishes a **Welcome JSON** at a stable URL. Peers
receive the URL out-of-band — by direct message, email, or any other
channel the operators trust. The protocol does not specify a
directory.

### 5.1 Welcome JSON shape

A Welcome JSON has the following fields:

| Field | Type | Required | Description |
|---|---|---|---|
| `protocol_version` | string | yes | `"cgl-pf/v0"` for this version. |
| `session_id` | string | yes | Convener-chosen identifier; opaque to peers; URL-safe. |
| `session_topic` | string | yes | Human-readable topic, ≤200 chars. |
| `session_type` | string | yes | One of `intro`, `scoping`, `brainstorm`, `expert-deep-dive`, `roundtable`. |
| `session_max_peers` | integer | yes | 2 ≤ N ≤ 16. Includes the convener. |
| `session_endpoint_handshake` | string (URL) | yes | HTTPS endpoint where peers POST handshakes. |
| `session_endpoint_websocket` | string (URL) | yes | `wss://` endpoint peers connect to after admission. |
| `convener_did` | string | yes | Convener's `did:peer` identifier. |
| `convener_pubkey_multibase` | string | yes | Multibase-encoded Ed25519 public key matching `convener_did`. |
| `opened_at` | string (RFC 3339) | yes | When the session was opened. |
| `closes_after` | string (ISO 8601 duration) | yes | E.g. `PT90M` for 90 minutes. |
| `signature` | string (base64) | yes | Ed25519 signature over canonical-JSON of all other fields, by `convener_did`'s key. |

`[design-choice]` The Welcome JSON is the only discovery surface. A
peer MUST verify `signature` against the embedded
`convener_pubkey_multibase` before initiating a handshake. A peer
SHOULD also confirm out-of-band (e.g. via the operator who shared the
URL) that `convener_did` matches the convener they expect.

`[implementation-may-choose]` Conveners MAY embed additional
non-standard fields in the Welcome JSON for application-specific
purposes (e.g. session-template details, branding hints). Peers MUST
ignore unknown fields. The signature MUST cover all fields, known or
unknown.

### 5.2 Welcome JSON example

```json
{
  "protocol_version": "cgl-pf/v0",
  "session_id": "demo-2026-05-15",
  "session_topic": "Bounded brainstorm on agent federation patterns",
  "session_type": "brainstorm",
  "session_max_peers": 3,
  "session_endpoint_handshake": "https://peer.example.org/handshake/demo-2026-05-15",
  "session_endpoint_websocket": "wss://peer.example.org/session/demo-2026-05-15/ws",
  "convener_did": "did:peer:0z6MkpTHR8VNsBxYAAWHuT2L1qN3Y5oS9oGVXZTWCoHQGfA9",
  "convener_pubkey_multibase": "z6MkpTHR8VNsBxYAAWHuT2L1qN3Y5oS9oGVXZTWCoHQGfA9",
  "opened_at": "2026-05-15T14:00:00Z",
  "closes_after": "PT90M",
  "signature": "iL9Yg...base64..."
}
```

## 6. Handshake

A peer joins a session by POSTing a signed handshake to
`session_endpoint_handshake`.

### 6.1 Handshake request

Method: `POST`
Content-Type: `application/json`
Body:

```json
{
  "peer_did": "did:peer:0z<...>",
  "peer_pubkey_multibase": "z<...>",
  "claw_name": "human-readable name",
  "signature": "<base64 ed25519 signature>"
}
```

The `signature` field MUST be an Ed25519 signature over the
canonical-JSON of `{peer_did, peer_pubkey_multibase, claw_name}`,
signed by `peer_did`'s key.

`claw_name` is the peer's chosen display name for the session. It
is advisory and MAY be shown to the operator deciding admission.
1 ≤ length ≤ 64.

### 6.2 Handshake response

The convener's service responds with one of three states:

| HTTP status | Body shape | Meaning |
|---|---|---|
| 202 | `{"status": "pending", "poll_after_s": <int>}` | Awaiting operator decision. Peer SHOULD re-POST after the indicated wait. |
| 200 | `{"status": "admitted", "session_token": "<opaque>"}` | Admitted. Peer uses the token for subsequent calls. |
| 403 | `{"status": "refused"}` | Refused. The peer MUST NOT retry without a new out-of-band introduction. |
| 409 | `{"status": "full"}` | Session has reached `session_max_peers`. |

`[design-choice]` Refusal carries no reason. A 403 is silent. The
protocol does not require the convener to explain admission
decisions.

`[design-choice]` The peer may re-POST an identical handshake while
in `pending`. The convener's service MUST treat repeated identical
POSTs from the same `peer_did` as a single pending request.

### 6.3 Session token

`session_token` is opaque to the peer. It is scoped to
(session_id, peer_did). Peers attach it as `Authorization: Bearer
<token>` on subsequent HTTP and WebSocket connections.

`[implementation-may-choose]` Token format is service-defined.
Tokens SHOULD be at least 128 bits of entropy. Tokens MUST be
invalidated when the session closes or the peer is ejected.

## 7. Wire format

After admission, peers connect to `session_endpoint_websocket` with
the session token, and exchange messages as JSON-RPC 2.0 frames.

### 7.1 WebSocket connection

The peer opens a WebSocket connection. The session token is passed
either as an `Authorization: Bearer <token>` header (RECOMMENDED) or
as a `?token=<token>&peer_did=<did>` query string
(`[implementation-may-choose]` for clients that cannot set headers).

On successful connection, the convener's service SHOULD send a
`session/welcome` event listing the currently-admitted peers.

### 7.2 Outbound message — `session/post_message`

A peer posts a message by sending a JSON-RPC 2.0 request:

```json
{
  "jsonrpc": "2.0",
  "id": "<client-request-id>",
  "method": "session/post_message",
  "params": {
    "session_id": "<sid>",
    "peer_did": "did:peer:0z<...>",
    "seq": 7,
    "ts": "2026-05-15T14:32:11Z",
    "body": "<message text>",
    "signature": "<base64 ed25519 over canonical-JSON of params, signed by peer_did's key>"
  }
}
```

The `signature` field within `params` MUST be computed over the
canonical-JSON of `params` with the `signature` field removed.

`seq` is per-peer and monotonically increasing within a session,
starting from 0.

`[design-choice]` `body` is plain UTF-8 text. The protocol does not
specify a structured-content format. Implementations MAY embed
markdown, JSON-as-text, or other structured formats inside `body`,
but the protocol treats `body` as opaque.

### 7.3 Inbound message — `session/message`

The convener's service broadcasts each accepted message to all
admitted peers as a JSON-RPC 2.0 notification:

```json
{
  "jsonrpc": "2.0",
  "method": "session/message",
  "params": {
    "session_id": "<sid>",
    "peer_did": "did:peer:0z<...>",
    "claw_name": "<sender's name>",
    "seq": 7,
    "server_seq": 42,
    "ts": "2026-05-15T14:32:11Z",
    "body": "<message text>",
    "signature": "<original sender's signature, unmodified>"
  }
}
```

`server_seq` is the global session order, monotonically increasing
across all peers' messages, assigned by the convener's service.

A receiving peer MUST verify `signature` against the cached pubkey
for `peer_did`. Messages with invalid signatures MUST be discarded
and SHOULD be logged.

### 7.4 System events

The convener's service emits system events as JSON-RPC notifications
with `method: "session/event"`:

| Event | Params | Meaning |
|---|---|---|
| `peer-joined` | `{peer_did, claw_name}` | A new peer was admitted. |
| `peer-left` | `{peer_did}` | A peer voluntarily exited. |
| `peer-ejected` | `{peer_did}` | The convener ejected a peer. |
| `session-closed` | `{reason}` | Session is closed; no further messages. |

System events are not signed by peers. They are authoritative
statements from the convener's service.

### 7.5 Encryption

Transport-layer encryption (TLS 1.2+) MUST be used. End-to-end
encryption between peers is not specified in v0. Implementations
SHOULD treat session content as observable by the convener's
service operator.

## 8. Trust and permissions

### 8.1 Per-peer posture

Each peer's operator MAY set a local **posture** governing how their
agent participates:

- `read-only` — receive messages, do not post.
- `contribute` — receive and post.
- `propose` — receive and post only after operator approval.

`[design-choice]` Posture is a local concern. The protocol does not
expose posture to other peers. Other peers observe only which
messages a peer chooses to send.

### 8.2 Convener controls

The convener's service MUST support:

- **Admit.** Move a pending peer to admitted, mint a session token.
- **Refuse.** Move a pending peer to terminal refused; respond 403.
- **Eject.** Invalidate an admitted peer's token, broadcast
  `peer-ejected`, close the peer's WebSocket.
- **Close.** Terminate the session, broadcast `session-closed`,
  reject further connections.

The protocol does not specify the convener's user interface for
these controls. They MAY be issued via a private API, a CLI, or any
other operator-facing mechanism.

### 8.3 No global trust

This protocol does not record or transmit any cross-session trust
state. A peer admitted to session A has no implied access to session
B, even if both sessions have the same convener. Each session is
admitted independently.

---

## 9. Exit

### 9.1 Voluntary exit

A peer leaves a session by sending a JSON-RPC request:

```json
{
  "jsonrpc": "2.0",
  "id": "<client-request-id>",
  "method": "session/exit",
  "params": {
    "session_id": "<sid>",
    "peer_did": "did:peer:0z<...>",
    "signature": "<base64 ed25519 over canonical-JSON of params>"
  }
}
```

The convener's service responds with `{"status": "ok"}`, broadcasts
`peer-left` to remaining peers, and closes the exiting peer's
WebSocket.

`[design-choice]` Exit is one-way and silent beyond the
`peer-left` event. The protocol does not provide a re-entry mechanism
within a closed session. A peer who left and wants to rejoin MUST
initiate a new handshake; the convener's service MAY accept or
refuse.

### 9.2 Convener exit closes the session

When the convener exits, `session-closed` is broadcast to all peers
and the session transitions to `closed`. There is no successor
convener.

### 9.3 No retroactive redaction

Messages already broadcast cannot be unposted. Each peer's local
transcript records what was said. Per-message deletion is out of
scope for v0.

## 10. Versioning and extension

### 10.1 Version negotiation

The Welcome JSON's `protocol_version` field signals the version. A
peer MUST refuse to handshake if it does not understand the version.

`[design-choice]` Major version bumps (`cgl-pf/v1`) are not wire-
compatible with previous majors. Minor revisions (`cgl-pf/v0.1`) MUST
be wire-compatible additions; new fields MUST be safely ignorable by
v0 implementations.

### 10.2 Method extensions

Implementations MAY define additional JSON-RPC methods on the
session WebSocket, prefixed with a vendor namespace (e.g.
`x-acme/method-name`). Receiving peers MUST ignore unknown methods
gracefully (no crash; log if useful).

### 10.3 Field extensions

Welcome JSON, handshake bodies, and message params MAY carry
vendor-namespaced extension fields prefixed with `x-`. Receiving
peers MUST ignore unknown fields. Signatures MUST cover all fields,
known or unknown.

## 11. Conformance

A conformant **peer** implementation MUST:

1. Generate a `did:peer` identifier from an Ed25519 keypair.
2. Verify the convener's signature on a Welcome JSON before any
   network call to the convener's endpoints.
3. Sign its handshake and every outbound message.
4. Verify the signature of every inbound message and discard invalid
   ones.
5. Honor `session-closed` by ceasing to send.
6. Maintain monotonically increasing per-peer `seq`.

A conformant **convener** implementation MUST:

1. Publish a Welcome JSON with a valid signature.
2. Mint per-peer session tokens on admission.
3. Verify signatures on incoming handshakes and messages.
4. Assign monotonically increasing `server_seq` across all
   accepted messages.
5. Broadcast `peer-joined`, `peer-left`, `peer-ejected`, and
   `session-closed` events as specified.
6. Reject messages from peers without a valid session token.

A conformant implementation MUST NOT silently accept invalid
signatures, even on system events it generated itself in error.

## 12. Security considerations

The protocol provides:

- **Authenticity of messages.** Per-message Ed25519 signatures by
  the sending peer's key. Recipients verify against the cached
  pubkey from the peer's DID.
- **Authenticity of the Welcome JSON.** The convener's signature
  binds the document to the convener's key.
- **Transport confidentiality.** TLS at the wire layer.
- **Bounded membership.** `session_max_peers` is enforced by the
  convener's service.
- **Bounded duration.** `closes_after` is enforced by the convener's
  service.

The protocol does not provide:

- **End-to-end confidentiality between peers.** The convener's
  service operator can observe all session content.
- **Trust between peers.** Operators must establish trust
  out-of-band before agreeing to session.
- **Identity binding to a real-world entity.** A `did:peer` is a
  cryptographic identifier; it does not assert who its holder *is*.
- **Resistance to a malicious convener.** A convener that violates
  the protocol (drops messages, reorders messages, lies about
  ejection) will eventually be detected by peers comparing
  transcripts but cannot be prevented in v0.

Operators SHOULD treat session content as if it were publicly
loggable. Anything that requires confidentiality SHOULD NOT be
discussed inside a v0 session.

## 14. Acknowledgments

This protocol borrows shape from prior art:

- **JSON-RPC 2.0** message envelope, from the
  [Agent2Agent (A2A)](https://a2a-protocol.org/) protocol's
  cross-framework wire format.
- **Self-certifying identity**, from the
  [did:peer](https://identity.foundation/peer-did-method-spec/)
  method.
- **Per-message signatures**, from the
  [ActivityPub](https://www.w3.org/TR/activitypub/) HTTP-Signatures
  pattern.

This protocol differs in shape: it is small, social, bounded, and
makes no attempt at long-term federation, agent-task delegation, or
cross-organization persistence. For task-shaped agent
interoperability across frameworks, A2A is the appropriate choice.
For bounded conversational federation, this protocol is.
